## บทที่ 1

Data Philosophy: ข้อมูลไม่ใช่สถานะ แต่คือพยานหลักฐาน


### 1.1 ปัญหาของ State-Based System

State-Based System
หมายถึงระบบที่เก็บเฉพาะ “สถานะล่าสุด” เช่น:

```ts
{
"unitId": "ODA-01",
"location": "GRID-A1",
"status": "READY"
}
```

ระบบแบบนี้ดูง่าย แต่มีปัญหาใหญ่:

| ปัญหา | คำอธิบาย |
| --- | --- |
| Lost History | ไม่รู้ว่าสถานะนี้มาจากเหตุการณ์ใด |
| Silent Overwrite | ข้อมูลใหม่เขียนทับข้อมูลเก่าโดยไม่มีร่องรอย |
| Weak AAR | Replay เหตุการณ์จริงไม่ได้ |
| Poor Conflict Handling | ข้อมูลชนกันแล้วระบบอาจเลือก last-write-wins |
| Audit Weakness | ตรวจไม่ได้ว่าใครเปลี่ยนอะไร เมื่อไร และเพราะอะไร |

ในระบบบัญชาการหรือระบบฝึกยุทธวิธี ปัญหานี้อันตรายมาก เพราะภาพสถานการณ์ล่าสุดอาจดู “สะอาด” แต่ซ่อนความผิดพลาดไว้ข้างหลัง


### 1.2 J-TRINITY Data Doctrine

J-TRINITY ใช้หลัก:

Event First, State Later
“บันทึกเหตุการณ์ก่อน แล้วค่อยสร้างสถานะตามหลัง”

หมายความว่า:

สิ่งที่เกิดขึ้นจริงต้องถูกบันทึกเป็น Event

Event ต้อง append-only

State เช่น unitStatus หรือ COP view ต้อง derive จาก timeline

หากมีข้อมูลขัดแย้ง ต้องสร้าง conflict event

หากต้องแก้ ต้องสร้าง correction event

ห้ามแก้ event เดิม

นี่คือหัวใจของ Event-Sourced Architecture


### 1.3 Event Sourcing คืออะไร

Event Sourcing
อ่านว่า: อีเวนต์ ซอร์สซิง
หมายถึงสถาปัตยกรรมที่เก็บ “ทุกการเปลี่ยนแปลง” เป็น event แทนการเก็บเฉพาะ state ล่าสุด

ตัวอย่าง:

แทนที่จะเก็บแค่ว่า:

```ts
{
"unitId": "UNIT-A",
"location": "GRID-C9"
}
```

ระบบต้องเก็บลำดับเหตุการณ์:

```ts
[
{
"eventType": "FIELD_REPORT",
"unitId": "UNIT-A",
"location": "GRID-A1",
"timestamp": "2026-05-07T09:00:00.000Z"
},
{
"eventType": "FIELD_REPORT",
"unitId": "UNIT-A",
"location": "GRID-C9",
"timestamp": "2026-05-07T09:05:00.000Z"
},
{
"eventType": "CONFLICT_DETECTED",
"reason": "Same unit reported at different locations"
}
]
```

สถานะล่าสุดเป็น “ผลลัพธ์ที่คำนวณได้” ไม่ใช่ความจริงต้นทาง


### 1.4 ตารางเปรียบเทียบ State-Based กับ Event-Sourced

| ประเด็น | State-Based | Event-Sourced J-TRINITY |
| --- | --- | --- |
| สิ่งที่เก็บ | สถานะล่าสุด | เหตุการณ์ทั้งหมด |
| การแก้ข้อมูล | overwrite | correction event |
| AAR | จำกัด | replay ได้ |
| Conflict | มักถูกเขียนทับ | ถูกบันทึกและ quarantine |
| Audit | อ่อน | แข็งแรง |
| AI Explainability | อ้างอิงยาก | อ้าง eventId ได้ |
| Offline Sync | เสี่ยง split-brain | deterministic replay ได้ |

## บทที่ 3

Immutable Timeline และ Merkle DAG


### 3.1 Immutable Timeline คืออะไร

Immutable Timeline
หมายถึงเส้นเวลาที่ “ห้ามแก้ย้อนหลัง”

หลักการ:

เพิ่ม event ใหม่ได้

ห้ามแก้ event เก่า

ห้ามลบ event เก่า

หากข้อมูลผิด ต้องสร้าง correction event

หาก conflict ต้องสร้าง conflict event

หาก replay ผิด ต้องสร้าง integrity event

ใน SIM-LAB ระบุชัดว่า PHASE 2 ต้องสร้าง immutable timeline service เช่น getTimeline(), appendEvent(), clearTimelineForTraining(), exportTimelineJson() และ rule คือห้าม mutate previous events; ทุก event ต้องมี sequenceNumber, timestamp, source, eventType, payload, trustScore, simulationOnly, advisoryOnly


### 3.2 Merkle DAG คืออะไร

Merkle DAG
อ่านว่า: เมอร์เคิล แด็ก
ย่อจาก Merkle Directed Acyclic Graph
หมายถึงโครงสร้างข้อมูลที่เชื่อม event ด้วย hash เพื่อพิสูจน์ว่า timeline ไม่ถูกแก้ย้อนหลัง

แนวคิดง่าย ๆ:

event แต่ละตัวมี hash ของตัวเอง

event ใหม่เก็บ hash ของ parent

ถ้า event เก่าถูกแก้ hash จะเปลี่ยน

hash ที่เปลี่ยนจะทำให้ chain ทั้งหมดผิด

ระบบจึงตรวจจับ tampering ได้


### 3.3 Canonical Serialization

Canonical Serialization
หมายถึงการแปลง object เป็น string แบบ deterministic เพื่อให้ hash ออกมาเหมือนเดิมเสมอ

กฎพื้นฐาน:

object keys ต้อง sort ตามตัวอักษร

array order ต้องคงเดิม

undefined ต้องไม่ถูก serialize แบบสุ่ม

timestamp ต้องเป็น ISO string

ห้ามใส่ random runtime field ใน hash

hash ต้องไม่ขึ้นกับเครื่องที่รัน

## บทที่ 4

Trust Engine: วิศวกรรมความน่าเชื่อถือ


### 4.1 Trust ไม่ใช่ความจริง

Trust Score
คือคะแนนความน่าเชื่อถือของข้อมูล ไม่ใช่ “ความจริงแน่นอน”

ข้อมูลที่ trust สูงอาจยังผิดได้
ข้อมูลที่ trust ต่ำอาจยังจริงได้

ดังนั้น Trust Engine ไม่ได้ตอบว่า:

“ข้อมูลนี้จริงหรือไม่”

แต่ตอบว่า:

“ข้อมูลนี้ควรถูกใช้งานด้วยระดับความระมัดระวังเท่าไร”


### 4.2 Trust Formula พื้นฐาน

เอกสาร SIM-LAB กำหนด Trust Engine แบบง่าย:

trustScore =

### 5.5 Conflict Resolution Doctrine

J-TRINITY ห้ามใช้ last-write-wins เป็นกฎหลัก

Last-write-wins
หมายถึงระบบเลือกข้อมูลล่าสุดทับข้อมูลเก่าโดยอัตโนมัติ

timestamp อาจไม่เที่ยง

ข้อมูลลวงอาจตั้งใจส่งมาทีหลัง

AAR จะเสียความจริงของลำดับเหตุการณ์

เอกสาร J-NEXUS ระบุว่าเมื่อ offline queue จากหลาย node replay พร้อมกัน ห้ามใช้ last-write-wins อย่างเดียว ต้องใช้ deterministic resolver โดยพิจารณา authorityLevel, serverTimestamp, observedAt, eventId และ conflictStatus

## สรุปท้ายภาคที่ 3


## Key Takeaways

J-TRINITY เป็น Event-First System
ระบบต้องบันทึกเหตุการณ์ก่อน แล้วค่อย derive state

operationalTimeline คือ Source of Truth
unitStatus, COP, dashboard, AAR เป็น projection เท่านั้น

Global Event Envelope คือมาตรฐานหลักฐานกลาง
ทุก event ต้องมี source, timestamp, trust, conflict, rationale และ simulation/advisory flags

Merkle DAG ป้องกันการแก้ประวัติย้อนหลัง
hash chain ทำให้ timeline ตรวจสอบ integrity ได้

Trust Engine ต้องอธิบายได้
trust score ต้องมาจากปัจจัยที่เห็นได้ ไม่ใช่ตัวเลขลอย ๆ

Conflict Engine ต้อง preserve contradiction
conflict ไม่ใช่ noise แต่เป็นสัญญาณว่าความจริงยังไม่สมบูรณ์

ห้าม last-write-wins เป็นกฎหลัก
โดยเฉพาะใน DIL/offline replay

Codebase Convergence คือเงื่อนไข production
one schema, one queue, one timeline, one trust model, one guardian gate

Server-Enforced Write Path ต้องมาแทน Client Direct Write
production ต้องผ่าน validation, policy, trust, guardian และ audit

ทุกอย่างยังคง SIMULATION_ONLY / ADVISORY_ONLY / HUMAN_FINAL
ไม่มี data architecture ใดมีสิทธิ์ทำลายกฎเหล็กนี้


## ประโยคปิดภาคที่ 3
